# Semantic Conventions for Physical AI

[![Status](https://img.shields.io/badge/Status-Draft-orange.svg)]()
[![OTel Version](https://img.shields.io/badge/OTel_Spec-v1.24.0-green.svg)](https://opentelemetry.io/)

This repository defines the open-source **Semantic Conventions for Physical AI**. It provides a standardized framework for mapping machine learning decisions, simulated intent, physical execution, and real-world deviations within the [OpenTelemetry (OTel)](https://opentelemetry.io/) standard.

---

## 💡 Why Semantic Conventions for Physical AI?

Traditional observability tools (like OpenTelemetry, Datadog, or Dynatrace) are designed for pure software stacks: HTTP requests, database transactions, and microservice RPCs.

However, in the world of **Physical AI**—where Artificial General Engineers (AGEs) and models make decisions that translate into real-world actions (e.g., controlling a CNC machine, programming a robotic arm, or designing hardware parts)—errors can lead to catastrophic hardware failure or safety hazards.

If a machine fails or deviates during testing, engineering teams need a **"Black Box" (Flight Data Recorder)** to audit:
- Was the failure caused by a thermodynamic simulation error?
- Was it a wrong material selection by the model?
- Did the model issue a faulty CNC instruction?
- Or did the hardware joint suffer from mechanical backlash/wear?

These semantic conventions solve this problem by providing a structured vocabulary to trace the entire path:
`Engineering Prompt` ──► `Simulation Results` ──► `Material Choices` ──► `Mechanical Commands` ──► `Physical Feedback (Deviations/Anomalies)`

---

## 🏗️ Attribute Namespace Design

The conventions are divided into four core namespaces under OpenTelemetry:

| Namespace | Attribute Prefix | Purpose |
| :--- | :--- | :--- |
| **Model & Decision** | `ai.model.*`, `ai.decision.*`, `ai.simulation.*` | Tracks model names, engineering prompt hashes, simulation validation status (kinematics, thermodynamics), and design parameters (like material choice). |
| **AI Intent** | `ai.intent.*` | Identifies high-level target goals and plans generated by the model. |
| **Physical Control** | `ai.physical.*` / `physical.sensor.*` | Tracks mechanical instructions issued and the target device or actuator. |
| **Action & Anomaly** | `physical.action.*` / `physical.anomaly.*` / `physical.test.*` | Measures expected vs. actual states, mechanical anomalies, corrective overrides, and research test session IDs. |

See [SPECIFICATION.md](./SPECIFICATION.md) for the full list of attributes, types, and requirements.

## 🚀 Getting Started

### 1. Ingesting Traces

Traces carrying these semantic conventions can be ingested by any OpenTelemetry-compatible collector or gateway. If you are using the **PhysiTrace** ecosystem, the ingest worker automatically extracts these attributes to index and display deviations in real-time.

```json
{
  "traceId": "4bf92f3577b34da6a3ce929d0e0e4736",
  "spanId": "00f067aa0ba902b7",
  "name": "Model Command Execution",
  "attributes": [
    { "key": "ai.model.name", "value": { "stringValue": "prometheus-age-v1" } },
    { "key": "ai.simulation.type", "value": { "stringValue": "thermodynamics" } },
    { "key": "ai.simulation.result", "value": { "stringValue": "passed" } },
    { "key": "ai.intent.goal", "value": { "stringValue": "high_precision_milling" } },
    { "key": "ai.physical.command", "value": { "stringValue": "SET_RPM" } },
    { "key": "physical.sensor.target", "value": { "stringValue": "cnc-spindle-04" } },
    { "key": "physical.telemetry.deviation", "value": { "doubleValue": 12.4 } },
    { "key": "physical.test.session_id", "value": { "stringValue": "session_20260619_01" } }
  ]
}
```

### 2. Visualization

Once ingested, you can easily build dashboard visualizers (such as in Grafana) that correlate AI confidence levels and simulation runs with real-world mechanical deviations, allowing you to catch drifting models before they damage physical equipment.
